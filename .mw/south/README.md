## Active Learning

### Evaluation Case Studies

Every MMOT evaluation teaches something. When veritas evaluates a DigitalModel — walks through its elements, scores current reality against desired outcome — patterns emerge:
- Which elements get honest assessments vs. which get inflated scores
- Where structural tension is named clearly vs. where it's buried in vague language
- How action plans either advance toward resolution or oscillate between comfort positions

These patterns are the curriculum. Each evaluation is a lesson in what makes assessment honest.

### Honest vs. Performative Evaluation

The sharpest distinction veritas is learning to make: the difference between an evaluation that *names what is* and one that *performs the appearance of naming*. Performative evaluation uses the right vocabulary — "current reality," "structural tension," "action plan" — but softens every finding. Honest evaluation sometimes hurts. It names the gap without qualifying it away.

The structural markers of performative evaluation:
- All elements rated 3+ out of 5 (everything is "acceptable")
- Action plans that restate intentions rather than changing behavior
- Tensions "resolved" by renaming them as features

### The Veritas / Coaia-Narrative Distinction

A critical learning edge: veritas evaluates **DigitalModels** (decision-making and performance review structures). Coaia-narrative evaluates **STC charts** (structural tension charts with current reality, desired outcome, and action paths). These are different objects requiring different evaluation logics.

The temptation is to merge them — "evaluation is evaluation." But the objects differ:
- DigitalModels have elements with flags and scores
- STC charts have narrative arcs with resolution patterns
- MMOT applies differently to each

This distinction protects both tools from becoming generic.

### Learning from Siblings

Veritas doesn't learn in isolation:
- **coaia-narrative MCP** teaches patterns of tool exposure — how evaluation capability becomes invocable
- **mia-code CLI** teaches patterns of integration — how tools compose in a developer's workflow
- **medicine-wheel ontology** teaches the type system — how DigitalModel, DigitalElement, and evaluation types stay coherent across repos

Each sibling implementation is a mirror showing veritas what it might become — and what it must not dilute.
